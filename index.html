Configurando Git por Primera vez

Al ingresar un comando y no retornar ningun error, significa que todo va bien.

--Global significa que configuras para todas las carpetas de tu sistema operativo. En un repositorio se pueden tener mas de dos cuentas 


    git --version
    git config --global user.name "Ing. Leonel Villa"
    git config --global user.username "l2v2blue"
    git config --global user.password "lv..2021"
    /* Email con el que tiene la cuenta vinculada con github */
    git config --global user.email l2v2blue@gmail.com
    /* Activar los colores en la UI de la terminal */
    git config --global user.ui true
    git config --global init.defaultBranch main
    /* Muestra en pantalla la configuración, salir con letra q */
    git config --list
    # asignar visual studio code como editor de configuración de git
    git config --global core.editor "code --wait"
    # Abre editor (VScode) con configuracion generada (Global y local) y espera a que el editor cierre el archivo
    git config --global -e
    # para estandarizar los saltos de línea en windows
    git config --global core.autocrlf true
    # para estandarizar los saltos de línea en linux/mac
    git config --global core.autocrlf input
    # ver todas las opciones de la configuración en la terminal git <comand> -h
    git config -h
    # ver todas las opciones de la configuración en el navegador git help <comand>
    git help config

Inicializar Git en un directorio local

Crear o desplazarse hasta la carpeta donde se inicializará git


  mkdir carpeta
  cd carpeta
  touch README.md     opcional contiene información y documentación del repositorio
  touch .gitignore    archivos que no participan, oculto, listar con ls -a
  # inicializar como proyecto para gestionar con git, generando un directorio oculto .git 
  git init
  code .

Flujo Básico

Flujo de Git, consta de tres estados locales y uno de forma remota cuando accedemos al codigo centralizado: modified, staged, committed y remote a cada uno le corresponde un área de trabajo.

  1. Working Directory: Area correspondiente al estado modified, es la carpeta local de PC donde se almacenan los archivos del proyecto.
  2. Staging Area: Es el área correspondiente al estado staged también se le llama index por que es el área donde git indexa y agrega los cambios realizados en los archivos previos a comprometerlos en su registro. INDEX | STAGING. Estado de indice o estado de ensayo.
  3. Local Repository: Es el área correspondiente al estado committed, donde los cambios ya se han registrado en el repositorio de git también se le llama HEAD o Local por que indica en qué cambio se encuentra el puntero del repositorio.
  4. Remote Repository: Es el área correspondiente al estado remote y es el directorio remoto donde almacenamos los archivos del proyecto en alguna plataforma web como GitHub, GitLab, BitBucket. Git denomina origin al repositorio remoto. ( GitHub | Remote ) 

    # agregar los cambios de un archivo al staged
    git add archivo/directorio
    # agregar todos los cambios de todos los archivos al staged
    git add .
    # Descartar cambios en working directory antes de pasar al Staging
    git restore <file>

    # los cambios son comprometidos en el repositorio y debe escribir el mensaje del cambio cuando se abra el archivo de configuración al terminar guarde y cierre el archivo para que los cambios tengan efecto
    git commit
    # es un shortcut del comando anterior escribes y confirmas el mensaje del cambio en un sólo paso
    git commit -m "mensaje descriptivo del cambio"
    # Descartar cambios para unstage y regresarlo al Working directory
    git restore --staged <file>

    # se agrega el origen remoto de tu repositorio de GitHub
    git remote add origin https://github.com/usuario/repositorio.git
    git remote add origin https://github.com/l2v2blue/resumir_git.git
    # la primera vez que vinculamos el repositorio remoto con el local master
    git push -u origin master
    # para las subsecuentes actualizaciones, sino cambias de rama, en caso de otra rama entonces git push -u origin nombre-rama-diferente
    git push

    #para descargar los cambios del repositorio remoto al local
    git pull

Formas de ejecutar add
$ git add . /* Agrega todos los cambios */
$ git add file1.txt *.java file2.txt
$ git add nom_path1 nom_path2/*.js nom_path3
$ git add nom_path1 nom_path2/*.js nom_path3
Descartar cambios en working directory antes de pasar al Staging
$ git restore <file>

Formas de ejecutar commit
$ git commit -m "Referencia de las modificaciones contenidas
$ git commit -am "Referencia de las modificaciones contenidas con add . incluido"
# no ejecutar posterior a una Fusión mediante Manual Merge por sobreescritura de commit --amend
$ git commit -a --amend -m "Referencia de las modificaciones contenidas con add . incluido"
$ git commit --amend --no-edit
$ git commit -a --amend --no-edit
$ git commit -a --amend -m "Cambiando mensaje de última confirmación"
# Descartar cambios para unstage y regresarlo al Working directory
$ git restore --staged <file>

Crear la rama o cambiar de rama a la main
$ git branch -M main

Ejecución de push para subir o empujar los cambios hacia la branch (rama) del repositorio remoto, previamente se debe haber agregado el origen remoto del repositorio de GitHub en caso contrario agregar mediante
$ git remote add origin https://github.com/nom_usuario/nom_repositorio.git
La url se la obtiene de la cuenta github para el usuario y la url amigable letras, numeros y guión medio unicamente.
$ git remote add origin https://github.com/l2v2blue/resumir_git.git Particularizado

$ git push -u origin main
$ git push para subsecuentes actualizaciones sino hay cambio de rema.

Para obtener, jalar o bajar los cambios desde el remoto hacia el local
$ git pull

Para realizar cualquier cambio en el codigo de desarrollo previamente se debe ejecutar un $ git pull para obtener todas las modificaciones antes de modificar código.

Master a Main

Repositorios nuevos


      git init
      git add .   Del working directory hacia el Staging (estado escenificado) aplicará rastreo (tracked) inicialmente archivos se encuentra untracked (no rastreados)
      # Seleccionar los archivos a mover hacia Index | Staging. No pasan todos solo los cambios que seleccionemos
      git add file1.txt *.java file2.txt
      git commit -m "Primer commit" Pasaran los archivos a confirmados (commit)
      git commit -am "Primer commit"  Realiza un add . y un commit
      /* Crear la rama main y volverla principal en lugar de la master */
      git branch -M main
      git remote add origin https://github.com/usuario/repositorio.git
      git push -u origin main

Repositorios existentes


      /* Crear la rama main y convertirla en principal en lugar de la master */
      git branch -M main
      git remote add origin https://github.com/usuario/repositorio.git
      git push -u origin main

Reemplazar la rama master por main en GitHub


    # Paso 1 :: Crear rama local main y pásar historial de rama master con el comando, el cual crea la rama main y con -m pasa el contenido de master a main.
    git branch -m master main
    
    # Paso 2 :: Hacer un push de la nueva rama local main en el repositorio remoto de GitHub
    git push -u origin main
    
    # Paso 3 :: Cambiar el HEAD actual a la rama main
    git symbolic-ref refs/remotes/origin/HEAD refs/remotes/origin/main
    
    # Paso 4 :: Cambiar rama default de master a main en repositorio de GitHub.
    En el repositorio de GitHub selecconar Settings
    En Default Branch cambiar el nombre de la rama de master a main o a cualquier otra rama definida.
    Para más información consulte el enlace
    
    # Paso 5 ::  Elimina la rama master del repositorio remoto
    git push origin --delete master

Para reemplazar la rama master por main en Git

Se observa en la séptima linea de codigo del tema Configurando Git por Primera vez como se establece el --global para que el init.defaultBranch sea el main. 


            git config --global init.defaultBranch main

Ignorar archivos

En el archivo .gitignore se debe incluir todo lo que NO se quiera incluir en el repositorio. Se lo puede crear manualmente o con gitignore.io. 


    # esto es un comentario
    archivo.ext
    carpeta
    /archivo_desde_raiz.ext
    # ignorar todos los archivos con extensión exe 
    *.exe
    # ignorar todos los archivos que terminen en .log
    *.log
    # excepto production.log
    !production.log
    # ignorar los archivos terminados en .txt dentro de la carpeta doc,
    # pero no en sus subcarpetas
    doc/*.txt
    # ignorar todos los archivos terminados en .txt dentro de la carpeta doc
    # y también en sus subcarpetas
    doc/**/*.txt

Ramas

Una rama permitirá aislar una nueva funcionalidad en el código que después se podrá añadir a la versión principal.

Los cambios rastreados (tracked) en una rama, confirmados (commit) y luego al realizar un git push se debe ejecutar como git push -u origin nombre-de-la-rama, caso contrario genera error, donde --set-upstream se equivale a -u 


          fatal: The current branch html has no upstream branch.
          To push the current branch and set the remote as upstream, use

              git push --set-upstream origin html

          To have this happen automatically for branches without a tracking
          upstream, see 'push.autoSetupRemote' in 'git help config'.

Crear rama

git branch nombre-rama

Cambiar de rama

git checkout nombre-rama

Crear una rama y cambiarse a ella

git checkout -b nombre-rama

Eliminar rama

git branch -d nombre-rama

Eliminar rama método forzado

git branch -D nombre-rama

Eliminar ramas remotas

git push origin --delete nombre-rama

Listar todas las ramas del repositorio

git branch

lista ramas no fusionadas a la rama actual

git branch --no-merged

lista ramas fusionadas a la rama actual

git branch --merged

rebasar ramas

git checkout rama-secundaria
git rebase rama-principal

Fusiones

Une dos ramas. Para hacer una fusión se necesita:

  1. Situarse en la rama que se quedará con el contenido fusionado.
  2. Fusionar.

Cuando se fusionan ramas se pueden dar 2 resultados diferentes:

  • Fast-Forward: Fusión se hace automática, no hay conflictos por resolver. Ramas trabajan en archivos diferentes.
  • Manual Merge: La fusión hay que hacerla manual, para resolver conflictos de duplicación de contenido. En cada rama afectan a un mismo archivo, se debe editar el archivo resultante del espacio Ej. main/merged y luego realizar un git add y un nuevo git commit para luego realizar un git push

Posterior a la fusión merge se requiere realizar un git push -u origin main o a la rama resultante de la fusion 


    # Cambiar a la rama principal resultante que quedará de la fusión
    git checkout rama-principal
    Ej.: git checkout main

    # ejecutar el comando merge con la rama secundaria a fusionar con la rama principal
    git merge rama-secundaria

    # Ejecutar un push de la rama principal donde queda la fusion
    git push -u origin rama-principal

    # Si se requiere eliminar la rama-secundaria posterior al merge o fusión
    git push -u origin --delete rama-secundaria

    # sin editar el mensaje de la última confirmación ( commit ) no ejecutar posterior a una Fusión mediante Manual Merge por existir un --amend, si existiera el commit en el push entonces se debe realizar pull y modificar manualmente similar a Fusion Manual Merge. Es recomendable usar --amend mientras no se haya ejecutado un push. Ni usarlo en la resolución de conflicto main/merged
    git commit --amend --no-edit

Si existiera el commit en el push entonces se debe realizar pull y modificar manualmente similar a Fusion Manual Merge. Es recomendable usar --amend mientras no se haya ejecutado un push. Ni usarlo en la resolución de conflicto main/merged
    git commit --amend -m "nuevo mensaje para el último commit" 

    # eliminar el último commit     02:51:21

Podemos desplazarnos en el historial del repositorio hacia atrás o adelante en cambios o ramas , sin afectar el repositorio como tal.

Registro del Historial

git log nos permite conocer todo el historial de un proyecto, con la información de la fecha, el autor y id de cada cambio. 


    git log

    # muestra en una sola línea por cambio
    git log --oneline

    # guarda el log en la ruta y archivo que especifiquemos
    git log > commits.txt

    # muestra el historial con el formato que indicamos
    git log --pretty=format:"%h - %an, %ar : %s"

    # cambiamos la n por cualquier número entero y mostrará los n cambios recientes
    git log -n

    # muestra los cambios realizados después de la fecha especificada
    git log --after="2019-07-07 00:00:00"

    # muestra los cambios realizados antes de la fecha especificada
    git log --before="2019-07-08 00:00:00"

    # muestra los cambios realizados en el rango de fecha especificado
    git log --after="2019-07-07 00:00:00" --before="2019-07-08 00:00:00"

    # muestra una gráfica del historial de cambios, rama y fusiones
    git log --oneline --graph --all

    # muestra todo el registro de acciones del log
    # incluyendo inserciones, cambios, eliminaciones, fusiones, etc.
    git reflog

    # diferencias entre el Working Directory y el Staging Area
    git diff

Reseteo del historial

Podemos eliminar el historial de cambios del proyecto hacia adelante con respecto de un punto de referencia. 


    #nos muestra el listado de archivos nuevos (untracked), borrados o editados
    git status

    # borra HEAD
    git reset --soft

    # borra HEAD y Staging
    git reset --mixed

    # borra todo: HEAD, Staging y Working Directory
    git reset --hard

    # deshace todos los cambios después del commit indicado, preservando los cambios localmente
    git reset id-commit

    # desecha todo el historial y regresa al commit especificado
    git reset --hard id-commit

Resetear un repositorio

Si en algún momento tienes la necesidad de resetear el historial de cambios de un repositorio para que quede como si lo acabarás de crear ejecuta esta serie de comandos: 


    cd carpeta-repositorio
    mv .git/config ~/saved_git_config
    rm -rf .git
    git init
    git branch -M main
    git add .
    git commit -m "Commit inicial"
    mv ~/saved_git_config .git/config
    git push --force origin main

Etiquetas

Con esta opción git nos permite versionar nuestro código, librería o proyecto. 


    # listar etiquetas
    git tag

    # crea una etiqueta
    git tag numero-versión

    # eliminar una etiqueta
    git tag -d numero-versión

    # mostrar información de una etiqueta
    git show numero-versión

    # sincronizando la etiqueta del repositorio local al remoto
    git add .
    git  tag v1.0.0
    git commit -m "v1.0.0"
    git push origin numero-versión

    # generando una etiqueta anotada (con mensaje de commit)
    git add .
    git tag -a "v1.0.0" -m "Mensaje de la etiqueta"
    git push --tags

GitHub Pages

gh-pages es una rama especial para crear un sitio web a tu proyecto alojado directamente en tu repositorio de GitHub.

  • URL del repositorio: https://github.com/usuario/repositorio
  • URL del sitio: https://usuario.github.io/repositorio

Para crear esta rama especial en GitHub ejecutamos los siguientes comandos: 


    git branch gh-pages
    git checkout gh-pages

    git remote add origin https://github.com/usuario/repositorio.git
    git push origin gh-pages

    # para descargar los cambios del repositorio remoto al local
    git pull origin gh-pages

Colaboración en GitHub

Para poder colaborar en proyectos alojados en GitHub necesitamos hacer uso de los forks y pull requests, herramientas que nos ofrece la plataforma para dicho objetivo.

A continuación describo el proceso de colaboración en GitHub.

  1. Forkea el repositorio en el que quieras colaborar, para hacerlo, sigue las instrucciones de este enlace .
  2. Una vez forkeado el repositorio en tu cuenta de GitHub, clónalo en tu equipo de cómputo.
  3. En el repositorio local hay que configurar los orígenes remotos de tu nueva copía para tener ambos remotos, los originales (origin) y los de tu copia, para hacerlo, sigue las instrucciones de este enlace.
  4. Crea una rama nueva en tu fork local para hacer tu colaboración, y sincrónizala con tu repositorio remoto, para hacerlo, sigue las instrucciones de este enlace.
  5. Configura tu repositorio para que acepté cambios (pull requests), para hacerlo, sigue las instrucciones de este enlace .
  6. Crea una pull request, para hacerlo, sigue las instrucciones de este enlace.
  7. Espera a que el dueño del repositorio original, acepte tus cambios.
  8. Una vez que acepten tu pull request, es recomendable que borres la rama en la que trabajaste el cambio y actualices tu repositorio forkeado, con los cambios del repositorio original.

Anexo un resumen de los comandos a ejecutar para colaborar en un repositorio de GitHub: 


    # forkear repositorio
    git clone https://github.com/usuario/repositorio.git
    git remote -v
    git remote rename origin fork
    git remote add origin https://github.com/usuario/repositorio.git
    git checkout -b rama-nueva
    git push fork rama-nueva
    # solicitar el pull request
    # aceptar el pull request
    git checkout main
    git pull origin main
    git push fork main
    git branch -d rama-nueva
    git push fork --delete rama-nueva
